Role Based Access Control

Role based access control is a methodology in which all users have a role. That role determines the access users have in a specific web app or service. (Check out ACL)

We can think of roles as classes where one role can inherit the attributes of others. We can see an example of this everywhere we have a form of tiered access. If we bring to mind a simple blogging web app. We could think of the author as a user with a role, a subscriber as a user with a different role, and a guest as another user with a different role. When each of them visits the site it makes sense for them to see different content. Through role based access control we make this possible.

Implementation wise we’d see the system working as follows.

  1. The client attempts to authenticate
  2. Assuming successful authentication, a web token is generated and returned with the users’ credentials / roles (signed)
  3. From then on the client adds the token to the authentication header of each request to the server
  4. The middleware processes the token and makes routing decisions based on the access the user has

The implementation of this traditionally follows the request pipeline where the backend receives the request, middleware handles the request, the backend does something, it returns a response to the client, and finally the frontend re-renders.
